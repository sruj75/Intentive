## What to build

Implement the **Mobile Client**'s Protocol WebSocket client boundary (`packages/protocol/`) so **Companion Chat** can send user messages, receive streaming assistant output, surface errors/retries, and report **Agent State** — without embedding **Agent Runtime** logic in UI code.

**Routing:** after Pre-Chat Gates, call **Control Plane** `GET /agent` for `ws_url` + `runtime_jwt`, then connect with the `connect` event. **Control Plane** is not on the message data path.

**Conversation Start Trigger:** the bootstrap opening is generated by the **Agent Runtime** after **Session Start** (Control Plane → Agent Runtime, internal). The **Mobile Client** must not author a hardcoded welcome message or a separate onboarding chat mode. Retrying a failed first opening must be idempotent at the product level (no duplicate openings in the UI).

A **test double** may implement the same boundary interface for automated tests and local UI iteration only. It is not a shippable alternate chat mode and must not bypass the Protocol contract.

## Acceptance criteria

- [ ] Protocol client exposes a narrow domain-facing contract: connect/disconnect, send `user_message`, observe inbound `companion_message` (and streaming/progress if added to Protocol), delivery status, error, retry, and Agent State.
- [ ] Production path uses `GET /agent` Routing then WebSocket events from `packages/protocol/` — not Control Plane message proxying.
- [ ] Sending a user message produces an assistant response through the Protocol boundary against a real or stubbed **Agent Runtime** gateway.
- [ ] First chat entry while relationship onboarding is incomplete waits for a runtime-generated opening after **Session Start** / **Conversation Start Trigger** — no client-authored bootstrap copy.
- [ ] The **Mobile Client** does not select a separate Relationship Onboarding chat mode.
- [ ] Initial-opening failure can be retried without rendering two openings.
- [ ] Loading, error, retry, and delivery status are exposed for #39/#40 UI.
- [ ] Agent State reports at least Available and Thinking from runtime-reported signals.
- [ ] UI does not imply on-device agent action; capability comes from Protocol-reported state.
- [ ] Contract tests cover send/receive, streaming/progress, error, retry, Agent State, first-opening delivery, and idempotent first-opening retry (test double allowed; no parallel dev chat product path).
