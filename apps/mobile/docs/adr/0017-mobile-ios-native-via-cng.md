# iOS native via Continuous Native Generation; drop committed native folders

Status: accepted

The Mobile Client previously committed both `ios/` and `android/` native projects to
git (a "bare" Expo workflow). We now **git-ignore both** and generate `ios/` on
demand with `npx expo prebuild` (Continuous Native Generation, CNG). `app.json` plus
config plugins are the single source of truth for all native configuration.

## Why

The committed native folders were **100% stock prebuild output with zero hand-edits**
— verified file by file:

- `ios/AppDelegate.swift` was the verbatim generated SDK 56 file (it still carried
  the `// Extension point for config-plugins` marker).
- `ios/Intentive.entitlements` held only `aps-environment`, which the
  `expo-notifications` plugin writes on every prebuild.
- `Podfile`, `Podfile.properties.json`, `ios/.gitignore` were all generated.
- There were **no custom config plugins** anywhere in the repo.
- Re-running `expo prebuild -p ios` reproduced the project identically (same bundle
  id `com.heyintentive.expo`, same `EXUpdatesRuntimeVersion 0.0.0`, same push
  entitlement).

So the folders carried **no information** that `app.json` didn't already hold — they
were derived output checked into source. Committing them cost us real friction:

1. **Bare-workflow restrictions.** EAS rejected the `runtimeVersion` _policy_ with
   _"bare workflow … runtime version policies are not supported"_, forcing a literal.
2. **`app.json` was silently ignored.** `expo-doctor` warned that with native folders
   present, EAS does not sync `scheme`, `ios`, `plugins`, etc. from `app.json` — two
   sources of truth that could drift.
3. **`android/` was a broken stub** — 17 files, only `mipmap-*` icons and a `values/`
   dir, no Gradle project. Unbuildable dead weight for an iOS-only product.

## Decision

- `apps/mobile/.gitignore` ignores `/ios` and `/android`.
- The `android/` stub is deleted; this is an iOS-only product (see
  [`../../AGENTS.md`](../../AGENTS.md)). Android, if ever needed, is regenerated by
  prebuild like iOS.
- `app.json` + config plugins are the **only** native source of truth. Native config
  changes go through `app.json` or a config plugin — **never** a hand-edit to `ios/`
  (it is regenerated and edits are lost).
- The dev and release flows gain a prebuild step:
  [`../DEVELOPMENT.md`](../DEVELOPMENT.md) (local dev-client simulator loop) and
  [`../RELEASE.md`](../RELEASE.md) (EAS Build prebuilds automatically when `ios/` is
  absent).
- We keep the literal `runtimeVersion` `"0.0.0"` (a policy is valid again under CNG,
  but a literal keeps the OTA compatibility key explicit and deterministic).

## Consequences

- **Single source of truth.** No more `app.json`-vs-native drift; the doctor warning
  is gone.
- **A prebuild step is now required** before a local build (and `pod install` runs as
  part of it). EAS Build does this on its own. The generated `ios/` is a regenerable
  local cache (treated like `node_modules`/`Pods`), not committed state.
- **Native changes must be expressed as config** (`app.json` / config plugin). A
  future engineer who needs a bespoke native edit writes a config plugin; if that is
  ever truly impossible, reversing this ADR (re-committing `ios/`) is the escape
  hatch — but the bar is "prebuild genuinely cannot express it."
- **Build inputs are re-fetched on a clean prebuild** (Pods, prebuilt RN
  xcframeworks). The fix is to **build in place and persist the outputs** (`pnpm
ios` reuses `ios/Pods` + DerivedData incrementally), not to re-commit the folders.
  Note `eas build --local` is structurally non-cacheable (fresh empty workingdir per
  run) and is reserved for portable/parity artifacts — see
  [`../DEVELOPMENT.md`](../DEVELOPMENT.md) §Build caching.
