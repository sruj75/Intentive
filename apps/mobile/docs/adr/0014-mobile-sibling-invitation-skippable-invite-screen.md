# Sibling Client Invitation is a skippable invite screen â€” the phone never claims the Mac connected

The **Sibling Client Invitation** (#21) is the **Pre-Chat Gate** after the **Consent Primer**: a skippable, capability-honest screen that invites the user to set up the **Desktop Client** so the companion gets fuller context. Its only first-party action is **"Not now"**, which writes `siblingInvitation: "skipped"` into **Launch State** via the store's `setSiblingInvitation` mutator; the resolver then advances to **Companion Chat**. The phone does not â€” and cannot â€” connect the Mac, so it never writes `completed` itself. Like #20 it is a **screen-only slice**: the `SIBLING_INVITATION_PENDING` destination, the `(gates)/invite` route, the `setSiblingInvitation` mutator, the `needs-invite` dev scenario, and `GateStatus`'s `skipped` value all already exist from #18; #21 only replaces the stub with the real screen.

**Considered Options** (central decision: what can the phone truthfully record?)

- Guidance + skip only (chosen): the screen offers an explainer and a "Not now" skip; a real `completed` is **server-observed** â€” the Mac registers through the Device Registry (#27) and `GET /me` reports it (#26). A `__DEV__`-only "mark connected" control exercises the completed path for tests/dev. The phone never asserts a capability it cannot verify (matches the PRD's capability-honesty rule and ADR 0011: the client owns no durable gate truth).
- A self-attested "I've set up my Mac" button writing `completed`. Rejected: the phone cannot verify a Mac connected, so the claim is capability-dishonest â€” it can mark the gate done with no sibling client present.

**Consequences**

- **Static guidance, no pairing machinery.** The screen is an explainer (what connecting the Mac improves) plus a plain pointer to where to get the Desktop Client. No QR, deep link, "email me the link", or account pairing â€” making _that_ Mac link to _this_ account is server/Device-Registry work (#27), not a launch screen's job.
- **No live detection in v1.** A Mac that connects while the screen is open is not watched for; the connected state is picked up on the next state resolve (`GET /me`, relaunch/refresh), consistent with hydrate-on-mount (ADR 0011). "Not now" already unblocks the user, so a launch gate never grows a realtime/polling backchannel (that would overlap the Protocol client, #33).
- **The "required/blocking" variant is deferred.** The issue's optional-vs-required capability state is not built. A required launch gate conflicts with mobile-PRD User Story 12 ("contextual prompts, _not_ a repeated blocking gate"); the honest handling of "you need the Mac for this experience" is contextual **in-chat** (#41), which #21's own out-of-scope confirms. The resolver and `GateStatus` are therefore unchanged â€” no "required" field, no new destination.
- **Skip persistence and recovery live elsewhere.** That a skipped eligible gate does not recur on ordinary relaunch is **#26** (durable Pre-Chat Gate state behind `GET /me`); the Mobile screen only writes optimistically. Initiating setup later after a skip is the **Account Surface**'s (#46). #21 must only avoid losing the status, which is already server-owned â€” nothing to build here. The skipâ†’advance and relaunch-after-skip mechanics are already proven by `resolve-launch-state.test.mjs` and the `needs-invite` / `ready` source scenarios; #21 adds no resolver test, only the screen test.
