# In-app silent auto-update on launch and wake

The Desktop Client auto-updates itself **silently** using Tauri's built-in updater (`tauri-plugin-updater`). It checks for a newer notarized build **on app launch and on macOS wake-from-sleep**, downloads and installs the update in the background without prompting, and the new version takes effect on the next launch/restart. There is no "update available" nag and no interruption of an in-progress Capture Session. Update artifacts (`.app.tar.gz` + detached `.sig`) and a `latest.json` manifest are published to the desktop GitHub Release; the updater endpoint points at the Release's `latest.json`. The updater signing key (`TAURI_SIGNING_PRIVATE_KEY`) is a CI-only secret; only its **public** key is embedded in `tauri.conf.json`.

## Considered Options

- **Silent background update _(chosen)_** — check, download, and install with no UI; user simply runs the newer version next launch. Lowest friction; the app can change between launches, which is acceptable for a capture-only menu-bar utility with no in-app chat state to disrupt.
- **Notify-and-let-them-restart** — show a quiet "Update ready — restart to apply" affordance in the menu bar and wait for the user. Rejected for v1: adds UI surface and a decision the user does not need to make for a background utility.
- **Embed/auto-restart immediately** — download and relaunch the app under the user. Rejected: relaunching could interrupt a live Capture Session.

### Check cadence

- **On launch + on wake-from-sleep _(chosen)_.** macOS lid-close **suspends** the process; it does not quit it. Reopening the Mac **resumes the same process** rather than launching a new one, so a launch-only check would never fire for users who only sleep/wake for days. Hooking the macOS wake notification makes "checks every time the user opens their Mac" literally true.
- **On launch only** — rejected: misses the sleep/wake case above (the common pattern), so long-lived installs would never update.
- **Periodic timer (e.g. every 6h)** — rejected as unnecessary: launch + wake already covers every time the machine becomes usable; a polling timer adds wakeups with no added coverage.

## Consequences

- **No nag UI and no mid-session interruption.** The installed update applies on the next launch/restart; a running Capture Session is never relaunched out from under the user.
- **Wake-from-sleep is a first-class trigger.** The updater scheduler must observe the macOS wake notification, not only `setup`/launch. This is the load-bearing detail that makes silent update actually reach long-lived installs.
- **The updater private key is a release-critical secret.** `TAURI_SIGNING_PRIVATE_KEY` lives only in GitHub Actions secrets and must be backed up out of band. Losing it means no future build can produce signatures that existing installs accept — a hard fork of the installed base. Rotating the embedded public key strands every prior install.
- **The first shipped version cannot self-heal a broken updater.** The usual bootstrap caveat: updater config errors in the first release can only be fixed by a fresh manual download. The release smoke (see `docs/RELEASE.md`) must therefore exercise a real N → N+1 update round-trip before the landing-page link is flipped.
- **Manifest custody is on the GitHub Release.** `latest.json` is generated by CI and attached to the desktop GitHub Release; the updater endpoint is the Release's `latest.json` URL. No separate hosting to maintain.
- A manual **"Check for Updates"** menu item is optional and out of scope for v1; the launch + wake triggers are sufficient.
